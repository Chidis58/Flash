Offline signal states — what feedback the sender receives when receiver is unreachable

state 1 — no internet, phone on

RX

Receiver has no data / wifi

Phone is on, app cannot be reached in real time

signal queued

Delivered the moment they reconnect. Sender sees "queued" tick — one faint tick, like SMS pending.

Sender knows: sent but not yet delivered. Not a failure — just waiting.

state 2 — phone off or unreachable

RX

Phone switched off or out of coverage

Signal cannot be queued at device level

signal held — phone unreachable

Server holds the signal. Delivered on first reconnect. Sender sees hollow tick with timestamp of when it was held.

Sender knows: not reached yet, but will be — with time context so they can decide whether to escalate.

state 3 — do not disturb / focus mode

RX

Receiver has DND or focus active

Phone reachable but notifications suppressed

signal held — DND active

Sender sees DND icon + estimated window end if receiver has shared their schedule. Signal queues for when DND lifts.

Sender knows: they're reachable but unavailable. Gives context so the sender respects the boundary.

state 4 — signal expires

RX

Signal held too long — context window closes

e.g. "come fix my hair" is irrelevant after 3 hours

signal expired

Each signal relationship has a configurable expiry window. Past that window, the signal is voided — not delivered. Sender gets an expiry notice.

This is critical. Delivering a stale "come now" hours later causes confusion. Context-expiry is a first-class feature, not an afterthought.

state 5 — delivered, no reply

RX

Signal delivered, receiver went silent

Seen but no got-it or decline within reply window

delivered — awaiting reply

After a configurable reply window (e.g. 5 mins), sender gets a "no reply" nudge. Option to re-ping or escalate — one tap.

Silence is not the same as offline. The sender deserves to know the difference.
